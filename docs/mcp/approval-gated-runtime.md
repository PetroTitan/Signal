# Approval-gated runtime

The MCP runtime enforces five distinct approval levels. The level is encoded on every operation type in `src/core/mcp-operations/operation-permissions.ts` and respected by every action that touches the runner.

## Levels

| approval_mode | UI gate | Examples |
| --- | --- | --- |
| `no_approval_needed` | Run button executes immediately | Safe reads, smoke tests, RLS check, env check |
| `approval_required` | Approve / Reject buttons | Apply a migration (request only), publish a draft, promote a screenshot-imported record |
| `explicit_text_confirmation_required` | Approve button only enabled after the operator types the deterministic phrase | Apply a migration, merge a PR, production redeploy, enable live execution |
| `blocked` | No approve button rendered; action refuses | Service-role exposure, external platform login, autonomous publish |

## Phase E2.6 additions

- `approveMcpOperationAction` now reads the run's `approval_mode` from the database (never trusted from the client) and refuses if:
  - the mode is `blocked`
  - the mode is `explicit_text_confirmation_required` and the typed phrase does not equal `approve production operation <run_id>`
- The Approve UI shows a confirmation-phrase input when the run's `approval_mode` is `explicit_text_confirmation_required`.
- The phrase is generated by `productionApprovalPhrase(runId)` so it is deterministic per run.

## What this protects

- **Merge a PR** — production-impacting. Will require phrase confirmation once the merge action ships.
- **Production redeploy** — same.
- **Enable live execution** — flipping `workspace_settings.execution_mode` from `dry_run` to `live` (future). Phrase confirmation required.
- **External publishing on any platform** — currently `blocked`. Phrase confirmation cannot override `blocked`.

## What this does NOT protect

- Local writes (drafts, prepared migrations) — these need a normal approve, not the phrase.
- Read-only checks — they execute on click.
- The smoke pipeline — it's `no_approval_needed` because it is read-only by construction.

## Audit

Every approval / rejection / phrase mismatch produces:

- A row update on `mcp_operation_runs` (`status='approved'` / `'rejected'`, `approved_by`, `approved_at`).
- An activity event (`mcp.<operation_type>` or the operation-specific event).
- Nothing in logs that contains the source text or phrase value beyond the operation summary.
